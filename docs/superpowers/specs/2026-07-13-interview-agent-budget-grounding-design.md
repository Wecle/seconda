# Interview Agent 预算、事实约束与消息分组设计

日期：2026-07-13

## 1. 背景与目标

面试 `fa7ba690-3d69-4c95-8a53-1bacf6b520ed` 的回答 Run 在候选人提交自我介绍后失败，界面显示“本轮处理达到最大步骤数，请重试”。同一页面还存在消息、思考摘要和状态卡归属不清的问题。

本设计只处理以下范围：

- 修复 Agent Run 在正常规划链路中提前耗尽预算的问题。
- 将规划工具调用、终局动作和供应商重试拆分为独立预算。
- 删除基于中文词面匹配的 `UNSUPPORTED_FACT` 校验。
- 保留 PRD 要求的简历证据存在性、题型上限、候选人回答轮次上限和确定性应用策略。
- 将面试消息按轮次分组并调整间距。

不修改六维评分模型、正式评分时机、面试状态机或最多 20 个候选人回答轮次。

## 2. 已确认根因

失败 Run `a737e9a5-c17a-4779-b877-fd2f58af855c` 的事件顺序为：

1. 读取最近面试历史。
2. 读取覆盖度。
3. 将自我介绍覆盖度更新为 `partial`。
4. 使用不存在的逻辑 ID 请求简历证据。
5. 改为读取 `resume:raw`。
6. 将自我介绍覆盖度更新为 `sufficient`。
7. 下一次模型调用开始前触发全 Run 供应商尝试上限。

该 Run 没有触发 `UNSUPPORTED_FACT`。错误来自以下预算和上下文设计共同作用：

- `MAX_MODEL_TURNS = 8` 将规划步骤和终局动作放在同一预算内。
- `MAX_PROVIDER_ATTEMPTS = 10` 在整个 Run 内累计，供应商重试会让 Run 在模型步骤上限之前失败。
- Prompt 已包含最近消息、覆盖度、轻量评估和简历证据目录，但 Skill 仍要求再次读取历史和覆盖度，产生冗余工具调用。
- 最近消息缺少可引用的消息 ID，模型为了获得 `answer:<messageId>` 必须调用 `get_interview_history`。
- 供应商尝试上限被映射为 `max_turns`，导致用户看到错误的失败原因。

## 3. 方案选择

### 3.1 采用方案

采用分层约束方案：

- 规划步骤最多 15 次。
- 提问或结束属于独立的终局动作，不计入 15 次规划预算。
- 终局动作参数修复最多 2 次。
- 供应商瞬时重试独立计数，不占规划步骤或终局修复次数。
- 删除本地中文词面蕴含判断，但保留来源 ID 存在性校验和确定性业务授权。

### 3.2 未采用方案

仅将原有总步骤数提高到 15 不足以解决问题，因为全 Run 供应商尝试上限仍可能提前触发，冗余读取和错误分类也会继续存在。

完整知识库式 claim/evidence/矛盾/时效系统不适合本次范围。OpenClaw 将这类 provenance-rich 能力放在独立 Memory Wiki 层；普通 Agent 召回仍以检索和上下文注入为主。Claude Code 的工具链重点校验 Schema、输入、权限和执行结果，并将工具结果回流模型，而不是对每句最终文本进行本地词面蕴含判断。

参考：

- https://github.com/openclaw/openclaw/blob/main/docs/concepts/memory.md
- https://github.com/liuup/claude-code-analysis/blob/main/analysis/04b-tool-call-implementation.md

## 4. 运行时预算设计

### 4.1 预算定义

运行时维护四组互不混用的计数：

| 预算 | 计数对象 | 上限 | 用途 |
| --- | --- | --- | --- |
| 规划步骤 | 成功或失败的非终局工具调用 | 15 | 读取证据、读取状态、更新覆盖度等规划行为 |
| 终局动作 | `ask_interview_question`、`finish_interview` | 不占规划预算 | 提交正式问题或结束语 |
| 终局修复 | 终局动作的可修复参数或业务错误 | 2 | 修复 Schema、证据 ID 或业务授权问题 |
| 供应商尝试 | 单次模型请求内部的模型候选和瞬时重试 | 每次模型请求局部有界 | 处理限流、超时和临时供应商错误 |

`final` 文本、未知工具和不可解析输出属于无效模型动作，不应伪装成成功规划。它们必须进入独立的无效动作保护，连续出现时终止 Run 并返回准确原因。

### 4.2 阶段切换

Run 分为两个运行时阶段：

1. `planning`：可以使用读取、覆盖度更新和终局工具。
2. `terminal`：只暴露 `ask_interview_question` 与 `finish_interview`。

满足任一条件时进入 `terminal`：

- 模型主动调用终局工具。
- 规划步骤达到 15。
- 确定性策略判定必须结束。

进入 `terminal` 后不可返回规划阶段。模型最多获得 2 次终局修复机会；修复预算耗尽后以 `terminal_action_failed` 结束，不再使用笼统的 `max_turns`。

### 4.3 供应商重试

供应商候选和最多两次瞬时重试保留在单次 `nextStepStream` 调用内部。运行时不再设置跨整个 Run 累加的 `MAX_PROVIDER_ATTEMPTS = 10`。

每次模型请求仍受局部重试上限、空闲超时和 AbortSignal 约束，因此删除全 Run 累加器不会产生无界重试。供应商链全部失败时使用 `provider_failed` 或现有等价的准确退出原因，不映射为 `max_turns`。

## 5. 上下文与工具调用设计

### 5.1 直接注入可用状态

Prompt 继续注入以下内容：

- 面试配置。
- 简历概览和证据目录。
- 当前覆盖度。
- 最近 8 条正式消息。
- 最新回答的轻量评估。
- 当前 Run 指令。

最近消息结构新增 `id` 和可选 `sourceId`：

- 用户消息：`sourceId = answer:<messageId>`。
- 面试官消息不生成 answer 来源 ID。

这样模型可以直接为候选人刚刚陈述的事实填写 `claims.sourceIds`，不需要先调用 `get_interview_history`。

### 5.2 Skill 指令

Skill 改为优先使用已注入上下文：

- 覆盖度已经在 Prompt 中时不得再次调用 `get_coverage_state`。
- 最近消息已经在 Prompt 中时不得再次调用 `get_interview_history`。
- 只有需要证据目录未包含的原文细节时才调用 `get_resume_evidence`。
- 不得使用自造的逻辑 ID；只能使用证据目录中出现的稳定 ID。

读取工具保留用于上下文压缩、目录信息不足或未来按需加载场景，不从工具注册表删除。

## 6. 事实约束设计

### 6.1 删除的约束

删除以下运行时链路：

- `validateGroundedClaims()` 对中文句子的词面重合、数字、拉丁词和二元组匹配。
- `validateGroundedResponse` 回调。
- `UNSUPPORTED_FACT` 工具错误。
- `UNSUPPORTED_FACT` 对应的修复预算消耗和测试。

该算法无法稳定判断语义蕴含，会把同义改写误判为无来源事实，并可能让本可继续的面试陷入修复循环。

### 6.2 保留的约束

以下约束继续由确定性代码执行：

- `resumeEvidenceIds` 必须存在于当前面试绑定的简历快照证据索引。
- `claims.sourceIds` 必须引用存在的简历证据 ID 或当前面试中的 `answer:<messageId>`。
- 问题必须关联至少一个合法简历证据 ID，除非动作是岗位澄清。
- 同题型最多 3 题。
- 全局最多 20 个候选人回答轮次。
- 重复问题、结束条件和用户主动结束由应用策略授权。

`claims` 继续作为内部审计信息，不显示给候选人。本地代码只验证来源是否存在和是否属于当前面试，不判断自然语言是否与来源逐词相似。

### 6.3 模型行为约束

Prompt 保留以下要求：

- 确定性事实必须填写来源 ID。
- 无法确认的事实改成询问句。
- 不补全年限、人数、技术栈、职责或成果。
- 候选人可见文本不得出现内部来源 ID。

这些约束降低幻觉风险，但不会以中文词面启发式阻塞整个 Run。

## 7. 错误分类与用户文案

退出原因与用户文案必须一一对应：

| 原因 | 用户文案 |
| --- | --- |
| 规划步骤达到 15 后终局动作成功 | 不显示错误 |
| `terminal_action_failed` | “本轮问题生成未能通过运行规则，请重试。” |
| `provider_failed` | “模型服务暂时不可用，请稍后重试。” |
| `blocking_limit` | “检测到重复处理，本轮已停止，请重试。” |
| `aborted_streaming` | 使用现有中断或连接恢复文案 |

前端优先展示 Run 终态事件中的 `userMessage`。刷新后从数据库读取终态时也必须得到相同文案，避免 SSE 与刷新路径不一致。

## 8. 消息分组与间距

采用按轮次分组的方案 A：

- 一个 Turn 包含候选人回答、该 Run 的公开思考摘要、状态卡和 provisional 面试官回复。
- 同一 Turn 内的块间距为 `12px`，对应 Tailwind `space-y-3`。
- 一个 Turn 与下一条正式面试官消息之间为 `28px`，使用明确的外层 `gap-7` 或等价布局。
- 面试官开场问题作为独立 Turn，与首个候选人回答保持较小的问答关联间距。
- 不再依赖 `display: contents` 配合列表级 `space-y-6`，因为该组合会把消息和 Run 附属块当成同级元素，削弱轮次归属。

渲染结构调整为显式 Turn 容器。Turn 容器负责内部间距，消息列表只负责 Turn 之间的间距。

## 9. 数据一致性

- 工具成功持久化后才显示状态卡，保持现有 PRD 规则。
- 已成功写入的覆盖度不因后续模型失败回滚。
- 同一 Run、同一 artifact 类型只显示一次；重复覆盖度更新不得生成重复“背景已保存”卡片。
- provisional 内容在 Run 失败时撤销，不进入正式消息记录。
- Run 的所有退出路径继续产生明确的 `run_completed` 或 `run_failed` 终态事件。

## 10. 测试设计

### 10.1 运行时单元测试

- 15 次非终局规划调用后进入 terminal 阶段。
- 成功的 `ask_interview_question` 和 `finish_interview` 不增加规划步骤计数。
- terminal 阶段只向模型暴露两个终局工具。
- 首次终局动作失败后最多允许两次修复，总计最多三次终局动作尝试；第三次仍失败时不再发起第四次模型调用。
- 单次模型请求的供应商重试不增加规划或终局修复计数。
- 供应商链失败不返回 `max_turns`。
- 重复规划工具仍受 loop detector 熔断。

### 10.2 事实约束测试

- 删除 `UNSUPPORTED_FACT` 相关测试。
- 同义改写的 acknowledgement 不再被本地词面算法拒绝。
- 不存在的 `resumeEvidenceIds` 继续返回 `EVIDENCE_NOT_FOUND`。
- 不存在或属于其他面试的 `claims.sourceIds` 被拒绝。
- 合法 `answer:<messageId>` 可以直接从 Prompt 最近消息中引用。
- 无简历证据的非澄清问题继续被确定性策略拒绝。

### 10.3 回归测试

使用失败 Run 的工具序列构造回归测试，验证即使先读取历史、读取覆盖度、读取两次证据并更新两次覆盖度，运行时仍能进入独立 terminal 阶段并提交问题，不会在第 7 次模型请求前以供应商尝试上限失败。

### 10.4 UI 测试

- 候选人回答、思考摘要和状态卡位于同一 Turn 容器。
- Turn 内使用 `space-y-3`。
- Turn 之间使用 28px 等价间距。
- 失败 Run 保持思考区展开，下一 Run 使用新的 Turn 分组。
- provisional 回复提交或撤销时不会改变相邻 Turn 的间距结构。

## 11. 验证命令

实现完成后运行：

```bash
pnpm lint
npx tsc --noEmit
pnpm test
pnpm build
```

并在本地浏览器复测指定面试页面，提交一次候选人回答，确认问题成功生成、错误文案正确且消息间距符合方案 A。

## 12. PRD 对齐

本设计符合完整 PRD 的以下要求：

- 问题基于面试简历快照，不虚构候选人经历。
- 模型只提出行动，题型上限、轮次上限、证据存在性和结束条件由确定性代码校验。
- 正式六维评分只在面试完成后执行。
- 同一题型最多 3 题，候选人最多回答 20 轮。
- Run 失败产生明确终态，provisional 内容失败时撤销。
- 用户消息、公开思考摘要、持久化成功后的状态区块按真实生命周期显示。
