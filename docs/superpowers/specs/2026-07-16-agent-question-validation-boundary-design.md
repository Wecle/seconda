# Seconda Agent 问题校验边界调整设计

**日期：** 2026-07-16

**状态：** 已批准

**范围：** 面试问题文本校验、模型提示、终结提案 Schema、失败修复与相关测试

**不改变：** 单 Agent loop、公开推理流式输出、PostgreSQL-only 事件链路、六维评分模型、简历快照、题型与轮次上限、完成条件

## 1. 背景

当前 Runtime 在结构化终结提案获得授权后，开始 provisional 流式展示 `responseText`，生成结束时再执行最终文本校验。`ask` 和 `clarify` 文本必须满足以下实现层规则：

- 全文恰好出现一个 `?` 或 `？`；
- 不能包含正则判定的多个疑问意图；
- 不能通过部分连接词追加问题或子问题。

这些规则并非 PRD 的业务要求。PRD 要求模型每轮只选择一个追问、新主题或结束行动，并由确定性代码限制分类、证据、轮次和完成条件；它没有要求候选人可见文本只能包含一个疑问句。

自然语言可以用一个主问题加解释、提示或拆解维度表达同一个考察目标。当前正则无法可靠区分“一个目标的多个提示”和“多个独立面试目标”，因此会把合法文本判为 `MULTIPLE_QUESTIONS`。由于文本已经 provisional 展示，这类误判进一步表现为内容撤回、重复生成和最终 Run 失败。

## 2. 产品定义

“一轮一个问题”统一解释为：

> 一轮只推进一个结构化面试行动、一个问题分类和一个核心考察意图；候选人可见文本可以包含多个疑问句、提示维度或辅助追问措辞。

唯一性由 `decision.action`、`decision.category`、`decision.intent` 和 `coverageTarget` 表达并由应用策略校验，不从自然语言标点或疑问词数量反推。

允许的表达示例：

> 你会怎样设计这条监控链路？可以分别说明采集、上报和分析怎么处理，以及为什么这样取舍。

该文本包含多个疑问表达，但只服务于“设计监控链路”这一核心考察意图，因此合法。

模型仍应避免在同一轮切换到两个无关主题，但这是生成指导，不是候选人文本的运行时正则校验项。

## 3. 方案选择

### 3.1 采用方案：结构化边界硬校验，问题措辞不校验数量

- 保留结构化行动、分类、意图、证据、覆盖目标和信息增益校验；
- 删除 `ask` / `clarify` 的问号计数与复合疑问句校验；
- 不再产生 `MULTIPLE_QUESTIONS`；
- Prompt 只要求围绕一个核心考察意图，不再要求一个疑问句或一个问号；
- 不增加固定 Planner、Renderer 或 Critic 模型调用。

该方案直接使文本规则与 PRD 对齐，同时保持现有 Agent loop 和实时流式体验。

### 3.2 未采用：保留正则并放宽阈值

允许两个问号或缩小复合句正则范围仍然需要从表面文本猜测语义边界，误判只是减少而不会消失，多语言维护成本也会持续增加。

### 3.3 未采用：增加独立模型审查

用第二次模型调用判断是否属于同一考察目标会增加延迟、成本和另一层非确定性。结构化提案已经拥有权威的行动与意图信息，没有必要从最终文案重新推断。

## 4. 校验边界

### 4.1 保留的确定性硬校验

- `decision` 每轮只有一个 action；
- `ask` / `clarify` 只有一个 category、intent 和 coverage target；
- evidence ID 属于当前不可变简历快照；
- 同一分类最多 3 题；
- 最多 20 个候选人回答轮次；
- Opening 不得由模型自行结束；
- Agent 自主结束满足轮次、分类、覆盖度或信息增益条件；
- `user_requested` 与 `max_rounds` 有持久化状态证明；
- 回复不泄漏内部协议、敏感信息或正式六维评分；
- 回复长度、面试语言和简历术语 grounding 满足现有约束；
- `finish` 文本不得继续邀请候选人作答，避免已完成状态和用户指令相互矛盾。

最后一项属于 action 一致性，而不是“一次几个疑问句”的风格限制。实现可以保留现有 `FINISH_ASKS_QUESTION` 防线及其有界修复路径。

### 4.2 删除的文本规则

- `ask` / `clarify` 必须恰好出现一个问号；
- 多个疑问词等同于多个问题；
- “以及”“并且”“然后”等连接词触发复合问题；
- 问题必须以问号结尾；
- acknowledgement 一律不得包含问号。

`responseText` 仍需为非空文本。问题是否适合当前轮次由结构化决策和 Prompt 保证，不再由标点保证。

## 5. 运行时与恢复

现有 Runtime 状态机和事件协议不变：

```text
reasoning
  -> proposal_streaming
  -> authorized
  -> responding
  -> validating
  -> committing
```

调整后，`responding` 阶段不会再因为问号数量或复合疑问句正则进入 `response_discarded`。安全、语言、grounding、正式评分泄漏等校验仍可能触发有界修复，因为这些失败代表真实的业务或安全风险。

不为问题措辞增加新的模型调用，也不增加确定性模板回放。现有实时 provisional 输出继续保留；本设计消除的是已确认不应成为失败条件的撤回来源。

Opening 继续使用现有结构化岗位判断分支，但 Prompt 从“必须且只能包含一个岗位方向澄清问题”调整为“只围绕岗位方向澄清这一核心意图”。Opening 不再因为辅助说明或多个疑问句失败。

## 6. 实现影响面

实现计划应覆盖以下位置：

- `lib/interview/agent/response-validator.ts`
  - 删除 `MULTIPLE_QUESTIONS` 结果码；
  - 删除 `ask` / `clarify` 的 `countQuestions` 和 `hasCompoundQuestion` 分支；
  - 若无其他调用需要，删除复合问句正则与相应辅助函数。
- `lib/interview/agent/grounding.ts`
  - 将 `singleQuestionSchema` 改为不校验问号数量的候选人提示文本 Schema；
  - acknowledgement 不再禁止问号。
- `lib/interview/agent/turn-proposal.ts`、`model-port.ts`、`skills.ts`
  - 删除“一个疑问句、一个问号、禁止连接词”等提示；
  - 改为“围绕一个结构化核心考察意图，不切换无关主题”。
- `lib/interview/agent/runtime.ts`
  - 删除 `MULTIPLE_QUESTIONS` 专用修复提示与分支。
- 相关单元测试与集成测试
  - 删除旧正则行为断言；
  - 增加多疑问句、多个提示维度仍可通过的回归用例；
  - 保留 finish、敏感内容、协议泄漏、语言、正式评分和 grounding 测试。
- PRD §7.2
  - 明确“一轮一个问题”的结构化语义，避免未来再次加入标点级硬约束。

## 7. 验收标准

1. `ask` / `clarify` 回复包含零个、一个或多个问号时，不会仅因此失败。
2. 同一回复出现多个疑问词或连接词时，不会产生 `MULTIPLE_QUESTIONS`。
3. 一个主问题附带采集、上报、分析等多个回答提示时可以正常提交。
4. 每轮仍只能提交一个结构化 action、category 和 intent。
5. 分类上限、轮次上限、证据、完成条件、语言、安全、正式评分和 grounding 校验保持有效。
6. Opening 不会再因疑问句数量撤回并重试。
7. 现有成功路径不增加模型调用。
8. Run、事件、SSE、恢复和 PostgreSQL-only 架构不发生变化。

## 8. 测试策略

- Validator 单测覆盖中文、英文、西班牙语和德语的多个疑问句文本。
- Grounding Schema 单测覆盖无问号陈述式邀请、单问号和多问号文本。
- Runtime 回归测试证明此类回复不会进入 repair，也不会产生 `response_discarded`。
- Opening 集成测试使用本次故障类型的多句澄清文本，验证一次提交成功。
- 保留所有非问题数量校验的现有负向测试。

## 9. 发布与兼容性

该调整不涉及数据库 Schema、迁移、环境变量、cutover 或历史数据回填。部署新代码后立即生效，不保留 V1/V2/V3 分支或灰度开关。

历史失败 Run 保持原始事实记录，不做篡改；用户重试时使用新规则执行。
